Node.js & Next.js Developer Assessment

Level 1 Assessment Level 1

Build a realistic admin-panel feature set on Next.js App Router with Prisma, RBAC, audit, and server-side validation. Includes a seeded intermittent bug you must debug.

Assessment overview

The Level 1 assessment verifies that the candidate can deliver a small but realistic admin-panel feature set end-to-end in a single AI-assisted session, and can reproduce and root-cause a seeded intermittent bug in the scaffolding they were given.

The assessment covers three connected parts: a bilingual announcement CRUD, a root-cause debugging task, and a role-aware user lookup with privacy-by-default. Every part must respect the business rules in the brief.

Scenario

You are working on the internal admin panel of a generic SMB SaaS product. Content managers publish bilingual announcements to end users. Support agents occasionally look up user information to help with tickets. The team has asked you to deliver three connected features in a single session, starting from a small scaffold repo that already includes authentication, a Prisma schema starter, and a shadcn/ui layout.

The scaffold contains one intentionally seeded intermittent bug in one of its existing routes that throws a 500 error under a specific edge case. Finding and fixing that bug at the root-cause level is part of the assessment.

What is required in the assessment

Part A — Bilingual announcement CRUD

  • App Router page at app/admin/announcements/page.tsx listing announcements with server-side pagination.
  • Create and edit pages with server-side Zod validation for English and Arabic title and body (both mandatory).
  • RBAC: only ADMIN and CONTENT_MANAGER may create, edit, or delete.
  • Every mutation writes an audit log entry via the provided helper.
  • Soft delete only. No hard deletes.
  • At least one negative test per mutation: unauthorised user blocked, invalid payload rejected, missing field rejected.

Part B — Root-cause an intermittent 500

  • Reproduce the seeded intermittent 500 in the scaffolding.
  • Identify the root cause. Do not hide the symptom.
  • Fix it in place and add a regression test that fails on the original code and passes on your fix.
  • Write a short POSTMORTEM.md covering: what was wrong, why it happened, how you reproduced it, how you prevented recurrence.

Part C — Role-aware user lookup

  • Server-side search and pagination for users at app/admin/users/page.tsx.
  • Support agents (role SUPPORT) see only display name, role, created date. Never email, phone, national ID, or any auth metadata.
  • Admins see the full record.
  • All queries go through Prisma — no in-memory filtering over the full table.
  • Include a database index on the searched columns.
  • At least one negative test: a Support agent cannot retrieve a sensitive field even by crafting the request manually.

Inputs provided

  • A starter scaffold repo: Next.js App Router, Prisma, User and Announcement schema, auth helper, RBAC helper, audit helper, Vitest harness, shadcn/ui layout.
  • A seeded intermittent bug in one of the scaffold's existing routes (Part B).
  • A one-page brief with business rules for Parts A and C and the symptom description for Part B.
  • Database credentials for a disposable PostgreSQL instance.
  • An AI tool of your choice. Any tool is allowed.

How to deliver the assessment

Delivery   Private GitHub repo. Branch assessment-1. Read access granted to reviewer account. Final commit pushed. Branch URL sent to reviewer.
  1. Create a private GitHub repository.
  2. Create a branch named assessment-1.
  3. Grant read access to the reviewer account you were given.
  4. Work through all three parts in one continuous AI session.
  5. Commit incrementally.
  6. As the final step, ask your AI tool to copy the entire session verbatim into ai-session/transcript.md with no modification.
  7. Push the final commit and send the branch URL to the reviewer.

Rules for applying this assessment

  • Mandatory One continuous AI session from start to finish. No multi-day work.
  • Mandatory Commit incrementally. A single final commit is a red flag.
  • Mandatory You are accountable for every line in the final repo.
  • Mandatory Identify the root cause of the seeded bug. Symptom-only fixes are forbidden.
  • Forbidden Post-hoc editing of the transcript. The transcript is the raw session.
  • Forbidden Bypassing RBAC or audit on any mutation.

  • Forbidden Hiding the Part B symptom instead of fixing the cause.

AI usage rule for this assessment

AI Rule   Any AI tool is allowed. Declare the tool in the transcript header. You are accountable for every line of code. A one-shot high-quality prompt is a winning pattern. Turn count is not measured.
  • Declare your AI tool name in the transcript header (Claude Code, Cursor, ChatGPT, Copilot, Claude CLI, or other).
